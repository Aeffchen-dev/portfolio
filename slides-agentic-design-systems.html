What happens when your design system grows a nervous system.

Design systems are built to scale consistency.
They rarely scale themselves.

Tokens drift. Components diverge. Documentation lags months behind reality. The system is always catching up — never ahead.

The idea

What if the design system
could notice, adapt,
and propose —
on its own?

Three shifts
  • 01 From docs to dialogue Ask the system a question. Get a grounded answer with token references and usage examples — not a stale wiki page.
  • 02 From static to generative Components as intent, not just files. An agent interprets a brief and scaffolds the right variant — constrained by the system, never outside it.
  • 03 From maintained to self-maintaining Agents that audit drift, flag broken token usage, and open pull requests to bring reality back in line with spec.
Tokens as language

Design tokens aren't just CSS variables.
They're the vocabulary agents use to reason about visual decisions.

When a token has semantic meaning — color.feedback.error instead of #f04 — an agent can make choices that are design-correct, not just technically valid.

All of this can be machine-readable. Most of it already is.

What agents can do
  • ↗ Audit — scan the codebase for off-token color or spacing values and surface them as issues
  • ↗ Generate — produce new components from a brief, grounded in your existing token set and patterns
  • ↗ Sync — keep Figma and code in agreement, flagging divergence instead of letting it compound
  • ↗ Document — write and update usage notes from actual component usage in the product
  • ↗ Advise — answer "which component should I use here?" with context from real patterns

Design system as living spec.
Product as constant signal.
Agents as the connective tissue between them.

The loop closes: what's built informs the system, the system informs what's built. No more divergence tax.

The real tension

Automation scales consistency.
Craft doesn't scale.

If agents handle all the maintenance work, does the designer stop understanding the system deeply? Does the system stop surprising us in useful ways?

The risk isn't that agents get things wrong. It's that they get things perfectly average.

Who owns the taste?

Agents can reason about rules. They can't yet feel when a rule should be broken for the right reason. That instinct — the one that makes a design system a design system and not a style guide — still lives with people.

The role shifts: from writing and enforcing the system to teaching and trusting it.

What I'm exploring
  • → A token schema designed to be agent-readable from day one
  • → An audit agent that runs on CI — no drift ships silently
  • → Figma ↔ code sync with an agent in the loop, not a person
  • → Taste gates — human review steps that can't be automated away

The design system of the near future isn't a tool you maintain.
It's a collaborator you shape.
